# 文案与 UX 设计规范

本文档定义了应用内的文案风格和用户体验（UX）设计原则。我们的目标是提供专业、友好且无摩擦的用户体验。

## ✍️ 文案风格 (Voice & Tone)

### 1. 语气原则

- **专业但不刻板**: 使用清晰、简洁的语言，避免过于正式的官僚用语，也避免过于随意的网络用语。
- **中立且客观**: 陈述事实，不带强烈的情绪色彩。
- **以用户为中心**: 使用“您”或直接省略主语，关注用户的需求和行动。

### 2. 禁忌词汇 (Do Not Use)

**严禁**在用户界面（UI）中暴露开发相关的技术术语或内部状态。

| ❌ 错误 (开发语言)               | ✅ 正确 (用户语言)         | 说明                                 |
| :------------------------------- | :------------------------- | :----------------------------------- |
| Session ID, Token                | 登录状态, 账号凭证         | 用户不关心技术实现                   |
| Debug, Stack Trace               | 错误详情                   | 仅在非生产环境或特定“高级详情”中展示 |
| Error 500, Internal Server Error | 服务器开小差了，请稍后再试 | 解释发生了什么，而不是错误码         |
| 杀死进程 (Kill Process)          | 强制结束, 停止             | 使用更温和的动词                     |
| 踢下线 (Kick Offline)            | 撤销登录, 退出设备         | 避免攻击性语言                       |
| Session Count                    | 在线设备数                 | 使用更易懂的指标                     |
| 提交 (Submit)                    | 保存, 确认, 发送           | 使用具体的动作描述                   |

### 3. 错误消息规范

错误消息应包含三个部分：

1.  **发生了什么**: 简明扼要地描述问题。
2.  **原因是什么** (可选): 如果用户能理解，简要解释原因。
3.  **怎么解决**: 最重要！告诉用户下一步该做什么。

**示例**:

- ❌ "Auth Failed: Invalid Token"
- ✅ "登录已过期，请重新登录。"

- ❌ "Network Error"
- ✅ "网络连接失败，请检查您的网络设置后重试。"

## 🎨 UX 设计原则

### 1. 反馈 (Feedback)

- **及时响应**: 用户的任何操作（点击、提交）都应立即得到反馈（Loading 状态、Toast 提示）。
- **非阻塞**: 尽量使用非模态（Toast, Inline Message）反馈，避免频繁弹出 Alert 阻断用户操作。
- **乐观更新 (Optimistic UI)**: 在服务器响应前先更新 UI，让应用感觉更快（需处理失败回滚）。

### 2. 加载状态 (Loading State)

- **骨架屏 (Skeleton)**: 优于旋转的 Spinner。骨架屏能让用户预知内容布局，减少感知等待时间。
- **渐进式加载**: 先加载核心内容，次要内容（评论、推荐）延迟加载。
- **防止布局偏移 (CLS)**: 为图片和动态内容预留固定宽高。

### 3. 导航与布局

- **一致性**: 保持导航栏、按钮位置和操作逻辑在全站的一致性。
- **面包屑**: 在深层级页面提供面包屑导航，方便用户返回。
- **空状态**: 当没有数据时，不要只显示空白，提供引导操作（例如：“暂无题目，点击右上角创建”）。

### 4. 移动端适配

- 确保所有功能在移动端可用。
- 触控目标大小至少 44x44 px。
- 避免横向滚动（除非是特定的 Carousel 或表格）。
